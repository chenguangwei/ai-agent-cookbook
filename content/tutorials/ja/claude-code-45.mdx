## Claude Codeをほぼ1年間使用した後、誰かが最も知る価値のある45のコツをすべてまとめているのを発見しました。

↓ 結論から言うと：ほとんどの人はその潜在能力の約30%しか使っていません。

───

🔋 **トークンの節約 / コスト削減**

Claude Codeのデフォルトシステムプロンプトは約20kトークンを消費します。作業を始める前からコンテキストの10%を消費していることになります。パッチ適用後は約9kまで削減でき、会話ごとに約10kトークンを節約できます。長期的な使用では差が大きくなります。

MCPツールについては、レイジーローディング（`ENABLE_TOOL_SEARCH: "true"`）を有効にすることをお勧めします。必要な時だけツールがロードされ、無駄にスペースを占有しません。

積極的に`/compact`を使用するか、`HANDOFF.md`を書いて新しいセッションに引き継ぎましょう。これは自動圧縮を待つよりもはるかに正確です。

───

🧠 **コンテキスト管理**

AIのコンテキストは牛乳のようなものです。新鮮なほど良いです。新しいトピックには新しいセッションを開始し、無理に継続しないでください。

超長い会話の場合、要約に頼らないでください。「セミクローニング」を使用します：会話の後半部分からの実際のメッセージのみを保持します。これにより、情報が歪むことなく保存されます。コンテキストが85%を超えたときに自動的にトリガーされるフックを設定できます。

───

🚀 **ワークフローの高速化**

音声入力はタイピングよりも速く、その差は顕著です。ローカルのWhisperモデルが時々文字起こしを間違えても、Claudeは理解できます。飛行機の中でも使えます。

マルチタスクには「カスケード方式」をお勧めします：3〜4個のターミナルタブを横に並べて開き、左から右へ順番にタスクを処理します。これにより、あちこち飛び回ることなく一方向に集中できます。

Git worktree：独立したディレクトリで複数のブランチを同時に作業し、互いに干渉しません。複数のタブと組み合わせれば、効率は倍増します。

ClaudeがRedditのようなサイトにアクセスできない？Gemini CLIをフォールバックとして設定し、tmuxを介してリレーすれば、調査タスクも実行できます。

───

🔍 **コード品質**

TDDはClaude Codeと驚くほど相性が良いです：まず、失敗するテストを書いてコミットし、次にClaudeにそのテストをパスするコードを書かせます。明確な目標があれば、その精度ははるかに高くなります。

GitHub Actionsのログを自分で調べる必要はありません。`/gha <url>`を使ってワンクリック分析：偶発的なフレークか、破壊的なコミットか、設定の問題かを自動的に判断し、正確な診断を提供します。

定期的に`cc-safe`を実行して、`settings.json`をスキャンし、`rm -rf`や`sudo`のような危険なコマンドを誤って承認していないか確認しましょう。

───

🤖 **上級編：マルチエージェント・オーケストレーション**

Claude CodeをDockerコンテナに入れ、`--dangerously-skip-permissions`で起動し、自律的に反復、修正、テストさせます。何か問題が発生しても、サンドボックス内に閉じ込められるので、ホストは安全です。

さらに進めて：外側のClaudeがtmuxを介してコンテナ内のワーカーClaudeを制御するようにします。コマンドを送信し、出力を読み取り、結果をプルする。すべて手動でターミナルを切り替えることなく。完全に自動化されたエージェントパイプラインです。

大規模なコードベースを並列で分析する必要がありますか？Claudeに複数のサブエージェントを同時に起動させ、異なるモジュールに作業を分担させます。タスクの複雑さに応じて計算リソースを割り当て、Opus / Sonnet / Haikuモデルを指定することもできます。

───

📁 **設定の習慣**

`CLAUDE.md`は空のファイルから始めましょう。繰り返し言っていることだけを追加します。簡潔に保ちます。定期的に見直し、古くなった内容は削除します。

ターミナルのエイリアスを設定：`c = claude`、`ch = claude --chrome`。1日に節約できる時間はバカになりません。

自動更新を無効にし（`DISABLE_AUTOUPDATER: "1"`）、新しいバージョンがシステムプロンプトのパッチを壊すのを防ぎます。
