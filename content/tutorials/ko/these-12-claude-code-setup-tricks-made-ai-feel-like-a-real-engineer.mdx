# 이 12가지 클로드 코드 설정 팁으로 AI가 진짜 엔지니어처럼 느껴졌다

대부분의 개발자는 클로드 코드를 더 똑똑한 ChatGPT처럼 사용하고 있다.

그게 실수다.

클로드 코드는 챗봇처럼 대하지 않을 때 진정으로 강력해진다.

...대신 AI 개발 환경으로 대우해야 한다.

가장 큰 전환점은 더 나은 프롬프트를 작성하는 것이 아니다.

모델 주변에 올바른 시스템을 구축하는 것이다.

대부분의 사람들은 클로드 코드를 설치하고 바로 이렇게 사용한다:

"이거 만들어줘"
"이거 고쳐줘"
"이거 리팩터링해줘"

하지만 진지한 개발자들은 먼저 환경을 최적화한다.

설정이 제대로 되면 모든 것이 시너지를 낸다:
- 더 나은 출력
- 더 깨끗한 컨텍스트
- 더 적은 환각
- 더 빠른 워크플로우
- 더 적은 정신적 부담
- 훨씬 더 나은 실행

다음은 내 빌드 방식을 완전히 바꾼 12가지 클로드 코드 설정 방법이다.

---

### 1. `CLAUDE.md`로 진정한 메모리 시스템 구축하기

대부분의 사용자는 채팅 기록에 전적으로 의존한다.

그건 신뢰할 수 없다.

고급 워크플로우는 대신 영구적인 프로젝트 메모리를 사용한다:
- 아키텍처 결정
- 코딩 패턴
- 디버깅 노트
- 엣지 케이스
- 제품 컨텍스트
- 반복되는 실수

클로드가 프로젝트의 실제 작동 방식을 기억하면 상호작용 품질이 완전히 달라진다.

매 세션마다 같은 내용을 다시 설명할 필요가 없어진다.

### 2. 새 코드베이스를 다루기 전에 `/init` 실행하기

이것은 가장 과소평가된 습관 중 하나다.

초기화 없이 클로드는 프로젝트에 거의 이해 없이 진입한다.

`/init`을 사용하면 매핑을 시작한다:
- 구조
- 의존성
- 규칙
- 워크플로우
- 프로젝트 패턴

출력 품질의 차이는 즉시 나타난다.

### 3. 병렬 AI 실행을 위해 Git 워크트리 사용하기

이것은 개발에 대한 사고방식을 바꾼다.

한 번에 하나의 AI 세션만 실행하는 대신, 여러 기능 브랜치를 동시에 격리할 수 있다:
- 인증 개선
- UI 재설계
- 버그 수정
- 실험

모두 메인 브랜치를 건드리지 않고 독립적으로 실행된다.

병렬 AI 워크플로우를 경험하면 일반 개발이 느리게 느껴진다.

### 4. 적절한 CLI 도구 설치하기

환경이 최적화되면 클로드의 능력이 크게 향상된다.

다음과 같은 도구들:
- `ripgrep`
- `fd`
- `jq`

이것들은 다음을 크게 개선한다:
- 파일 검색
- 검색 속도
- 파싱
- 디버깅

고급 AI 워크플로우의 큰 부분은 모델이 내부에서 작동할 더 나은 인프라를 제공하는 것이다.

### 5. 전략적으로 MCP 서버 사용하기

MCP는 클로드가 어시스턴트처럼 느껴지지 않고 실제 엔지니어링 시스템처럼 느껴지기 시작하는 지점이다.

훈련 데이터에만 의존하는 대신, 클로드는 다음과 상호작용할 수 있다:
- 라이브 문서
- 브라우저 도구
- 데이터베이스
- Notion
- API
- 디자인 시스템

이제 모델은 추측하지 않는다.

실제 외부 컨텍스트로 작동한다.

### 6. 터미널 전용 워크플로우에 제한하지 않기

많은 사람들이 순수 터미널 설정을 낭만적으로 생각한다.

하지만 클로드 코드를 VS Code와 함께 사용하면 훨씬 더 매끄러운 실행이 가능하다:
- 인라인 편집
- 더 나은 가시성
- 더 쉬운 탐색
- 더 빠른 반복
- 더 깨끗한 워크플로우

좋은 도구는 마찰을 제거한다.
그게 미학보다 더 중요하다.

---

### 7. 플러그인을 전문화된 AI 직원처럼 사용하기

대부분의 사용자는 기본 동작을 넘어서지 않는다.

플러그인은 그것을 완전히 바꾼다.

다음과 같은 집중된 워크플로우를 만들 수 있다:
- 프론트엔드 시스템
- 구조화된 기능 개발
- 정리/리팩터링
- 아키텍처 리뷰
- 문서 생성

하나의 일반 어시스턴트 대신 전문화된 운영자를 얻는다.

### 8. 재사용 가능한 슬래시 명령어 만들기

이것은 가장 효과적인 설정 개선 중 하나다.

프롬프트를 반복해서 다시 작성하는 대신 다음과 같은 워크플로우를 만든다:
- `/security-audit`
- `/optimize-query`
- `/generate-tests`
- `/review-architecture`

매번 수동으로 프롬프트를 작성할 필요가 없어진다.

워크플로우가 운영화된다.

### 9. 서브에이전트를 사용하여 컨텍스트 품질 보호하기

대부분의 AI 출력 품질은 컨텍스트가 오염되면서 무너진다.

서브에이전트는 이것을 아름답게 해결한다.

다음을 위해 격리된 에이전트를 생성할 수 있다:
- 코드베이스 연구
- 디버깅
- UX 분석
- 문서화
- 의존성 추적

그런 다음 유용한 결과만 다시 가져온다.

메인 컨텍스트는 집중되고 깨끗하게 유지된다.

### 10. 토큰 사용량을 진지하게 추적하기

대부분의 개발자는 비용이 폭발할 때까지 이것을 무시한다.

전문적인 워크플로우는 다음을 추적한다:
- 토큰 사용량
- 컨텍스트 증가
- 비용이 많이 드는 세션
- 불필요한 도구 호출

좋은 AI 엔지니어링은 부분적으로 지능에 관한 것이다...

...하지만 또한 리소스 관리에 관한 것이다.

### 11. 무거운 워크플로우를 위해 높은 토큰 제공자 사용하기

대규모 AI 코딩은 컨텍스트 제한이 사라질 때 변화한다.

방대한 할당량을 가진 모델은 다음을 가능하게 한다:
- 대규모 리팩터
- 거대한 저장소
- 다중 파일 추론
- 아키텍처 수준 계획

이것이 AI 코딩이 실험적으로 느껴지지 않고 산업적으로 느껴지기 시작하는 지점이다.

### 12. 클로드를 CI/CD에 직접 통합하기

이것이 진정으로 강력해지는 지점이다.

클로드가 다음과 같은 PR 워크플로우를 상상해보라:
- 코드 리뷰
- 수정 제안
- 표준 적용
- 아키텍처 규칙 준수
- 병합 전 문제 발견

이제 AI는 개발을 돕는 것이 아니다.

개발 라이프사이클 자체에 내장되어 있다.

---

대부분의 사람들은 AI 코딩이 코드를 더 빨리 작성하는 것이라고 생각한다.

그건 표면적인 사고다.

진정한 변화는 AI가 효과적으로 작동할 수 있는 시스템을 구축하는 방법을 배우는 것이다.

그것이 다음 사이의 차이다:
가끔 AI를 사용하는 것

vs

실제 AI 네이티브 엔지니어링 워크플로우를 구축하는 것.

그리고 솔직히?

대부분의 개발자는 여전히 그 격차가 얼마나 커지고 있는지 깨닫지 못하고 있다.
